# B端直返服务方式支持 - FDE交接文档

## 1. 文档目的

这份文档给产品侧 FDE/前端开发使用，只说明本期需要改的前端内容、后端就绪条件和验收方式。

最新范围以 2026-06-12 会议纪要为准：FDE 本期只做“服务类型管理页新增直返枚举列”，不做直返渠道管理页面，不改商品权益全套 CRUD 页面，也不新增前端接口。

## 2. 一句话背景

B 端政策需要新增一种服务方式“直返”。后端会把“直返”作为新的服务方式枚举和字典项提供出来，FDE 需要让服务类型管理页能展示、选择、保存和回显这个新选项。

## 3. FDE 承接范围

| 模块 | 页面 | FDE 要做什么 | 是否本期 |
| --- | --- | --- | --- |
| 服务类型管理 | 服务类型列表/编辑页 | 新增“直返”列或等价展示；编辑时可选择“直返”；保存后可回显 | 是 |
| 商品权益配置 | 商品权益新增/编辑/详情 | 不做页面改造，仅后端验证枚举兼容 | 否 |
| 直返渠道管理 | 新页面 | 不新增页面，不做查询/新增/编辑/删除/启停 | 否 |
| 政策增删改查 | 既有政策页面 | 不新增接口，不改主流程 | 否 |

## 4. FDE 不负责什么

以下内容不属于 FDE 本期任务：

1. 不新增直返渠道管理页面。
2. 不开发直返渠道查询、新增、编辑、删除、启用、禁用接口。
3. 不设计直返渠道菜单和权限点。
4. 不修改商品权益配置页完整 CRUD 流程。
5. 不修改政策增删改查接口。
6. 不处理工单履约、逆向仓、库存、财务链路。
7. 不准备直返渠道初始化数据。
8. 不发布 `xms-common` 或后端服务。

## 5. 前端涉及仓库和文件

| 仓库 | 文件 | 当前作用 | FDE 关注点 |
| --- | --- | --- | --- |
| `asp-operation-web` | `operation-front/src/views/policy/service-type/DATA.js` | 服务类型页面配置、列配置、服务方式字典请求 | 增加“直返”展示列或让列配置能识别新字典项 |
| `asp-operation-web` | `operation-front/src/views/policy/service-type/method.js` | 服务方式列表和 `way0...way17` 等页面字段映射 | 增加“直返”的映射和回显逻辑，避免数组下标错位 |

当前页面会请求服务方式字典，字典类型为 `base_service_way`。后端会在该字典中新增“直返”。

## 6. Git 和本地开发操作手册

### 6.1 仓库信息

| 项 | 内容 |
| --- | --- |
| 前端仓库 | `git@git.n.xiaomi.com:mit/after-sales/asp-operation-web.git` |
| 前端工程目录 | `operation-front` |
| 建议基线 | 远端最新 `master` |
| 建议开发分支 | `feature/FUZ-72-direct-return-service-way` |
| 本期建议改动文件 | `operation-front/src/views/policy/service-type/DATA.js`、`operation-front/src/views/policy/service-type/method.js` |

注意：请从远端最新 `master` 新建开发分支，不要直接基于本机未整理的本地 `master` 或历史实验分支开发。

### 6.2 拉代码和建分支

```bash
git clone git@git.n.xiaomi.com:mit/after-sales/asp-operation-web.git
cd asp-operation-web
git fetch origin
git switch master
git pull --ff-only origin master
git switch -c feature/FUZ-72-direct-return-service-way
```

如果已经有本地仓库，可以从 `git fetch origin` 开始执行，并确认 `git status -sb` 是干净状态后再新建分支。

### 6.3 安装依赖和启动本地服务

```bash
cd operation-front
npm ci
npm run start:proxy
```

说明：仓库中存在 `package-lock.json`，优先使用 `npm ci` 保持依赖版本一致；如果公司网络或私有包下载失败，按前端团队现有代理/registry 配置处理。

### 6.4 开发改动建议

建议只改服务类型管理相关文件，避免扩大范围：

```bash
operation-front/src/views/policy/service-type/DATA.js
operation-front/src/views/policy/service-type/method.js
```

开发时重点确认：

1. `DATA.js` 中“直返”列展示是否跟随现有服务方式列风格。
2. `method.js` 中“直返”映射是否来自后端字典项，而不是硬编码数组位置。
3. 保存接口提交的直返值是否为后端确认的 `6` 或现有接口结构中的等价值。
4. 刷新页面后“直返”状态是否可回显。

### 6.5 本地自测和构建

```bash
npm run build
```

自测时至少覆盖：

| 场景 | 检查点 |
| --- | --- |
| 页面打开 | 服务类型管理页正常加载，无控制台报错 |
| 字典返回 | 服务方式字典中包含“直返” |
| 列表展示 | 列表出现“直返”列或等价展示 |
| 编辑保存 | 勾选“直返”保存成功 |
| 保存回显 | 刷新后“直返”仍保持选中 |
| 旧功能回归 | 到家、到店、寄修等旧服务方式展示和保存不受影响 |

### 6.6 提交和推送

```bash
git status -sb
git diff -- operation-front/src/views/policy/service-type/DATA.js operation-front/src/views/policy/service-type/method.js
git add operation-front/src/views/policy/service-type/DATA.js operation-front/src/views/policy/service-type/method.js
git commit -m "feat: 服务类型支持直返服务方式"
git push -u origin feature/FUZ-72-direct-return-service-way
```

提交前确认不要带入无关文件，例如本地配置、构建产物、日志、临时截图等。

### 6.7 MR 信息建议

MR 标题建议：`feat: 服务类型支持直返服务方式`。

MR 描述建议包含：

1. 改动范围：服务类型管理页新增“直返”列和编辑选项。
2. 未改范围：不新增直返渠道管理页，不新增接口，不改商品权益页面。
3. 依赖条件：后端 `base_service_way` 字典已返回“直返”，服务方式值为 `6`。
4. 自测结果：列展示、编辑保存、刷新回显、旧服务方式回归、`npm run build` 结果。
5. 关联事项：关联 Multica/需求项 `FUZ-72` 或项目组确认的前端任务。

### 6.8 常见 Git 问题处理

| 问题 | 建议处理 |
| --- | --- |
| `master` 无法快进 | 先不要强制覆盖，确认本地是否有未提交改动；必要时重新 clone 干净仓库 |
| 私有包安装失败 | 检查 `.npmrc`、公司网络、SSH 权限和 npm registry |
| 分支名已存在 | 使用 `git switch feature/FUZ-72-direct-return-service-way` 或增加个人后缀 |
| MR 出现无关 diff | 用 `git diff --name-only origin/master...HEAD` 检查，只保留服务类型页相关文件 |
| 构建失败 | 先确认是否由本次改动引入；若为环境或历史问题，在 MR 描述中说明 |

## 8. 页面改造要求

### 8.1 列表页

服务类型列表需要能看到“直返”的支持情况。

| 项 | 要求 |
| --- | --- |
| 展示文案 | 统一使用“直返” |
| 展示位置 | 建议跟随现有服务方式列排列，放在旧服务方式之后 |
| 展示含义 | 当前服务类型是否支持直返 |
| 旧数据兼容 | 未配置直返的服务类型展示为未选中/否/空，按现有页面风格处理 |

### 8.2 编辑页

编辑服务类型时，服务方式区域需要出现“直返”选项。

| 项 | 要求 |
| --- | --- |
| 可选择 | 可以勾选/选择“直返” |
| 可取消 | 已选择后可以取消 |
| 可保存 | 保存请求能带上直返对应值 |
| 可回显 | 保存后刷新页面仍能看到直返状态 |
| 不影响旧项 | 到家、到店、寄修、厂家售后、虚拟服务受理等旧服务方式行为不变 |

### 8.3 接口要求

本期前端原则上不新增接口、不变更接口路径。

| 接口类型 | 处理方式 |
| --- | --- |
| 字典接口 | 继续使用现有 `base_service_way` 字典 |
| 服务类型查询 | 继续使用现有查询接口 |
| 服务类型保存 | 继续使用现有保存接口 |
| 直返渠道接口 | 不调用，因为本期没有渠道管理页面 |

## 9. 关键实现注意事项

### 9.1 不要把 `way18` 当成业务含义

历史页面可能把服务方式展开成 `way0`、`way1`、`way2` 等字段。若新增字段时命名为 `way18` 或其他序号字段，需要注意：

1. `way18` 只是页面内部字段名，不是“直返”的稳定业务标识。
2. “直返”的业务标识应以后端返回的字典值为准，例如 `serviceWay=6`、`code=ZF` 或文案“直返”。
3. 如果必须新增 `way18`，也要保证它是从“直返”字典项映射出来的，而不是简单认为数组第 18 个就是直返。

### 9.2 文案必须统一

页面文案统一使用：**直返**。

不要混用以下文案：

| 不建议文案 | 原因 |
| --- | --- |
| B端直返 | 页面服务方式枚举不需要带业务域前缀 |
| 生态链直返 | 范围比服务方式更具体，容易误导 |
| 直接返仓 | 不是本期确认的标准枚举文案 |
| ZF | 只适合作为代码/编码，不适合直接给运营展示 |

### 9.3 保存值必须和后端一致

| 项 | 值 |
| --- | --- |
| 服务方式 ID | `6` |
| 枚举编码 | `ZF` |
| 展示文案 | `直返` |

如果接口保存的是 ID，则提交 `6`；如果接口保存的是服务方式列表对象，则按现有结构增加直返对应项，不自行设计新结构。

### 9.4 不要顺手扩范围

不要因为看到“直返渠道表”就新增渠道管理页面。本期渠道表只由后端/DBA 初始化，不面向运营维护。

## 14. 常见问题

### Q1：需要新接口吗？

不需要。会议已确认前端只在服务类型管理页面新增“直返”枚举列，复用现有字典、查询和保存接口。

### Q2：需要做直返渠道管理页吗？

不需要。本期后端只建直返渠道表并做初始化准备，不提供渠道管理页面和增删改查接口。

### Q3：商品权益页面要改吗？

本期 FDE 不改商品权益页面。后端会验证服务方式枚举扩展、权益扩展查询字段和导入映射。

### Q4：如果页面现在写死 `way0...way17` 怎么办？

可以按现有结构新增一个页面字段，但必须从后端字典中的“直返”项映射出来。不要只依赖数组位置判断。

### Q5：上线后信息会变多还是变少？

页面信息会增加一个“直返”服务方式列/选项；接口本期不减少旧字段，权益扩展查询只新增可空字段，整体是向后兼容的“信息变多”。
