# 品类建单政策侧支持：需求分析与技术方案梳理

## 背景

服务小程序“用户自助品类建单”技术方案已评审到 v1.1（修订日期 2026-06-23）。总体方案采用“真品类建单 + 后续补充 SN/IMEI”：C 端建单阶段不再强依赖 goodsId，以 createType 新增品类建单枚举和 thirdCategoryId 作为主链路标识；不新增 orderSource；未补 SN/IMEI 前商品相关能力需要支持品类维度查询或空态/降级。

本任务承接“政策侧/配置侧”支持部分，目标仓库为 sfp-entitlement-config-service（git@git.n.xiaomi.com:mit/after-sales/sfp-entitlement-config-service.git）。本轮只做需求分析和技术方案梳理，不做代码实现。

## 已知主方案要点

1. 首期按二级类目灰度，覆盖空调（壁挂/立式/多联机）、冰箱、冰柜、洗衣机、电视、显示器。
2. 品类建单入口开关由现有分类树接口返回，三级品类列表复用现有分类树/商品列表接口。
3. C 端提交工单传 createType=品类建单新枚举、thirdCategoryId、thirdCategoryName、categoryImageUrl，不传有效 goodsId。
4. asp-aftersale-service 在建单、服务类型、派单、政策、取件、服务前准备、服务规范等查询链路支持 thirdCategoryId。
5. 履约平台/服务通展示待补充 SN/IMEI，补码后复用既有 SN/IMEI 识别真实商品流程刷新价格、三包、备件、保险等 SKU 依赖模块。
6. 未补 SN/IMEI 前，价格、三包、备件、保险等 SKU 强依赖模块展示空态或置灰，不允许因 goodsId 为空报错。
7. 运费提示本期倾向：品类建单未补 SN/IMEI 前不展示，接口直接返回空提示或 show=false；若业务需要品类维度费率，需要新增品类费率配置能力。

## 本轮需要分析的问题

请基于 sfp-entitlement-config-service 现有代码和配置模型，输出需求分析与技术方案建议，重点回答：

1. 政策/配置侧需要承接哪些能力？按 P0/P1/P2 拆分。
2. 现有配置模型是否天然支持 thirdCategoryId / brandClassId 维度？如果不支持，推荐新增字段、查询条件或适配层方式。
3. 服务类型、服务前准备、服务规范是否在 config 仓库内有对应配置/查询能力？需要如何支持品类维度。
4. 价格、三包、备件、保险、维修策略等 SKU 强依赖能力，哪些应由 config 仓库支持，哪些应由售后/履约侧做空态或补码后重算。
5. 运费、取件、闪送是否适合在本仓库增加品类规则/费率配置；如果不适合，应如何定义边界与降级。
6. createType 新枚举与“不新增 orderSource”的口径下，本仓库是否需要枚举/DTO/配置条件变更。
7. 给出最小可落地技术方案，包括数据契约、接口/服务改造点、配置迁移/初始化方式、灰度和回滚策略。
8. 给出验收标准和测试建议，不要求实现代码。
9. 列出需要业务/政策/售后侧确认的问题清单。
