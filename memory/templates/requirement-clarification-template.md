# 需求澄清模板

用于 `requirement-gate` 失败、用户请求过于模糊、或任务进入设计前缺少关键事实时。目标是用最少问题补齐可执行需求，不把需求沟通阶段变成方案设计。

## 使用原则

- 先问会影响方向、边界、验收和副作用的问题。
- 不一次性抛出所有问题；优先 3-7 个高价值问题。
- 能从上下文确定的事实不要重复问。
- 涉及飞书、Multica、远端 Git、部署、删除、生产访问时必须单独问审批和边界。
- 用户只给“继续/执行/推”时，如果需求仍缺核心验收或副作用边界，不得直接执行。

## 快速澄清格式

```markdown
我需要先确认以下信息，避免直接开工后返工：

1. 背景/问题：当前要解决的具体问题是什么？
2. 目标/验收：完成后你希望看到什么可验收结果？
3. 范围/边界：本轮只做哪些内容，明确不做哪些内容？
4. 依赖/输入：相关 repo、文档、任务 ID、接口或数据源是什么？
5. 副作用：是否允许飞书/Multica 写入、远端 Git、部署、删除或生产访问？
6. 时间：这轮希望先交付一个多大的切片？
```

## 按任务等级提问

### S0：问答/解释/轻量判断

- 期望输出是结论、建议，还是要落文档？
- 是否需要引用本地文件或外部链接？
- 是否需要我继续执行后续动作？

### S1：本地文档/脚本小改

- 目标文件或目录在哪里？
- 验收标准是什么：新增文档、脚本输出、还是验证命令通过？
- 是否允许修改现有规范、配置或模板？
- 是否需要同步 Obsidian generated？

### S2：代码/工具链/多文件改动

- 需求背景和非目标是什么？
- 影响哪些模块、命令、测试或用户流程？
- 需要保留哪些兼容行为？
- 验证命令和回滚方式是什么？

### S3：跨系统/外部写回/真实业务任务

- 任务事实源是哪个 Multica issue、飞书文档或业务需求？
- 写入目标、字段、评论位置和状态更新规则是什么？
- 审批范围、有效期和回读证据要求是什么？
- 如果写回失败或数据不一致，谁决策下一步？

## 输出骨架

```markdown
## 已知信息

- 背景：
- 目标：
- 范围：
- 非目标：
- 依赖：
- 验收：
- 副作用：

## 待确认问题

1. ...
2. ...
3. ...

## 暂停原因

当前缺少 <关键字段>，继续执行会导致 <具体风险>。

## 用户补充后下一步

补齐后我会先运行 requirement-gate，再进入设计/执行计划。
```

## 与门禁的关系

- `requirement-gate` 失败：使用本模板生成面向用户的问题。
- `design-gate` 失败：不要继续开发，先补方案和验证路径。
- `deliverable-gate` 失败：不要写回或宣称完成，先补证据、验证和下一步。
