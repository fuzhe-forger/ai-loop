# 方案模板偏好

## 基础技术方案默认模板

以后输出基础技术方案时，默认使用 `MIT 技术设计方案（架构+详细）标准&模版 V1.15` 的章节结构。

- 模板来源：`https://mi.feishu.cn/wiki/Wf2gw6EVsiija9k5AUqcKt8CntX`
- 本地快照：`memory/templates/mit-technical-design-v1.15.md`
- 适用范围：业务技术方案、架构+详细一体化方案、评审包初稿、基础方案草稿。

## 默认章节顺序

1. 注意
2. 修订版本
3. 设计概要
   - 背景
   - 设计范围
   - 设计难点
   - 术语定义
4. 架构设计/解决方案设计
   - 整体解决方案
   - 架构和关系图
   - 系统触发条件
5. 接口设计（对外提供的）
   - 接口协议
   - 接受的消息队列
   - 任务调度
6. 外部依赖（使用外部的）
   - 依赖的外部服务
   - 通过消息队列输出
7. 内部设计
   - 领域分解
   - 类图/交互图/序列图/流程图等
   - 高可用设计
   - 非功能特性设计
8. 数据设计
   - 数据安全性
   - 数据库和表设计
   - 缓存设计
   - 数据对接设计
9. 部署/运维
10. 高危元素
11. 影响范围
12. 可观测性
13. 风险评估
14. 时间线
15. 需求决策
16. 其他参考文档
17. 评审记录

## 使用规则

- 基础方案先按上述大章节组织，小章节可按项目特点裁剪或合并，但必须说明裁剪原因。
- 金额、库存、额度、权益核销、迁移、并发、权限、审批、外部系统依赖相关需求，必须保留 `高危元素`、`影响范围`、`可观测性`、`风险评估`。
- 涉及数据表、历史数据、迁移、灰度时，必须保留 `数据设计` 和 `部署/运维`。
- 涉及跨系统调用时，必须明确 `接口设计`、`外部依赖`、`系统触发条件`。
- 方案初稿允许标注 `待确认`，但不能空章节；不确定的信息要落到 `需求决策` 或 `风险评估`。
- 输出给 Multica/Loop 的方案草稿，应同步生成本地 evidence：task、stage-report、review-packet 或 design-draft。
