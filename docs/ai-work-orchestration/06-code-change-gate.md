# 真实代码改动准入门禁

## 目标

在进入真实代码改动前，先判断任务是否适合交给 AI Loop 执行，避免从文档/脚本样板直接跳到高风险自动化。

这份门禁回答三个问题：

1. 什么样的代码改动可以进入试点？
2. 进入前必须具备哪些输入？
3. 哪些动作仍然必须人工确认？

## 准入原则

真实代码改动试点必须满足以下原则：

- 小范围：只影响一个清晰模块或一组辅助脚本。
- 可验证：有明确、可重复执行的本地验证命令。
- 可回退：改动可通过 git diff 清晰审查，不依赖不可逆外部状态。
- 低副作用：不触发生产系统、远端写库、部署、批量数据修改。
- 证据完整：能产出 summary、stage report、review packet。

## 推荐首批任务

优先选择：

- CLI 帮助信息补充。
- 本地脚本参数校验增强。
- 文档生成脚本或报告生成脚本的小改动。
- 本地测试补充或测试数据样例优化。
- 不改变业务路径的开发者体验改进。

暂不选择：

- 生产故障修复。
- 鉴权、支付、订单、数据删除等高风险业务路径。
- 跨仓库大重构。
- 需要远端服务或真实账号验证的改动。
- 没有验收标准的探索性改动。

## 必备输入

进入代码改动前，任务文件必须包含：

- `Repo`：目标仓库路径。
- `Scope`：允许修改的文件或目录。
- `Out of scope`：明确不允许改的内容。
- `Acceptance criteria`：验收标准。
- `Verification command`：本地验证命令。
- `Approval policy`：哪些副作用允许自动执行，哪些必须人工确认。
- `Rollback note`：如何通过 git diff / git restore 回退。

## 执行流程

### 1. 先 dry-run

先使用 dry-run 或本地任务草案确认任务结构完整，不把 dry-run 成功视为业务完成。

### 2. 再真实改动

只有当任务满足准入原则、验证命令明确、工作树风险可控时，才进入真实代码改动。

### 3. 每轮都验证

每轮改动后必须运行最小相关验证，再根据结果决定是否扩大验证范围。

### 4. 生成复核包

完成后生成：

- `summary.md`
- `stage-report.md`
- `evidence-checklist.md`
- `review-packet.md`
- `multica-comment.md`

### 5. 人工复核后再远端动作

即使代码改动本地验证通过，以下动作仍需人工确认：

- `git commit`
- `git push`
- MR 创建或合并
- 部署
- 生产访问
- 数据库写入
- 删除或不可逆操作

## 状态策略

代码改动任务默认使用 `conservative`：

- dry-run 通过：保持 `todo`。
- 真实 run 通过且验证命令通过：可以进入 `in_review`。
- 验证失败：进入 `blocked` 或继续本地修复。

状态同步必须引用本地证据包和验证结果。

## 准入检查清单

进入真实代码改动前逐项确认：

- [ ] 任务范围小且明确。
- [ ] 修改文件范围已列出。
- [ ] 不修改生产配置、密钥、发布流程。
- [ ] 有本地验证命令。
- [ ] 验证命令预计不会访问生产系统。
- [ ] 远端副作用已声明。
- [ ] 回退方式清晰。
- [ ] 完成后能生成 review packet。

如果任一项无法确认，应停留在规划或文档阶段。
