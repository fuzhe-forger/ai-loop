# AI 工作编排领域模型

## 领域定位

本实践的领域不是“Multica 集成”，也不是“脚本自动化”，而是：

> Autonomous Work Orchestration，自主工作编排。

它关注一个工作项如何从意图进入系统，经过执行、验证、反馈，最终形成可判断的证据。

## 核心业务问题

我们需要回答几个问题：

- 一个 issue 是否已经清楚到可以执行？
- 如果不清楚，谁来规划？
- 如果可以执行，应该交给哪个能力角色？
- 执行是否发生在安全边界内？
- 结果用什么证明？
- 失败后下一步是什么？
- 什么时候需要人类决策？

这些问题构成了这个领域的核心。

## 边界上下文

### Work Intake

负责把外部任务变成系统内部可处理的工作项。

输入可以来自 Multica issue、人工输入、定时巡检或其他系统。第一阶段只处理单个 Multica issue。

核心职责：

- 读取 issue。
- 保存原始快照。
- 提取标题、描述、状态、优先级、负责人。
- 判断是否具备进入 Loop 的基本条件。

### Loop Execution

负责把工作项转成本地可执行任务，并调用 ai-loop 执行。

核心职责：

- 生成 task.md。
- 确定 repo、运行模式和验证方式。
- 调用 plan、dry-run 或 run。
- 收集 run.json、summary、diff、verify 结果。

### Evidence Management

负责保存和解释执行证据。

核心职责：

- 把 artifacts 当成执行事实来源。
- 从 summary 和 verify 结果中提取结论。
- 判断是否足以进入复核。
- 为失败生成下一轮输入。

### Feedback Sync

负责把证据转成控制台和人能理解的反馈。

核心职责：

- 生成 Multica comment 草稿。
- 在显式授权时写入 comment。
- 在显式授权时变更 issue 状态。
- 记录本地 run 与远端 issue 的映射。

### Policy and Safety

负责保护系统边界。

核心职责：

- 默认只读远端。
- 默认只本地执行。
- 外部写入必须显式授权。
- 批量处理必须显式开启。
- 高风险动作必须进入人工决策。

## 核心概念

### Work Item

一个进入系统的工作单元。它不等同于 Multica issue，而是 issue 在本地编排系统中的表达。

### Intent

工作项背后的目标。它回答“为什么要做”。

### Plan

执行前的方案。它回答“准备怎么做”。

### Execution

一次具体执行。它回答“实际做了什么”。

### Evidence

执行证据。它回答“凭什么判断完成或失败”。

### Decision

基于证据做出的下一步判断：继续、停止、复核、阻塞、升级或交给人。

### Feedback

给控制台、人类和其他智能体看的结果表达。

## 角色分工

### 黑墙

调度与总控角色。关注任务是否应该进入 Loop、谁来处理、什么时候停止和升级。

### Codex-顾实

主力工程执行角色。适合明确的开发、修复、重构和本地验证任务。

### Xcode

二意见与复杂判断角色。适合架构争议、失败重试、复杂方案复核和死循环打破。

### 测真

验证与测试角色。适合读取 evidence 后做复核、补充测试和发布前验证。

### 人类

目标、边界、验收、风险和最终决策的拥有者。

## 关键不变量

- 没有明确 repo 的任务不能进入真实 run。
- 没有验证方式的任务只能 plan 或 dry-run。
- 没有 evidence 的任务不能自动标记完成。
- 没有显式授权不能写远端状态。
- 失败不是终点，而是下一轮 Loop 的输入。

## 第一阶段领域切片

第一阶段只实现最小闭环：

```text
Issue -> Work Item -> Task -> Loop Dry-run -> Evidence -> Comment Draft
```

这条链路能跑通，就证明领域模型可以落地；后续再扩展为真实执行、回写、状态同步和队列调度。
