# 分享路线图：从工具到 AI 工作编排实践

## 分享目标

这个实践方向需要同时服务工程落地和团队认知升级。

每一阶段的分享不只是汇报“做了什么”，而是回答：

- 这个阶段解决了什么工程问题？
- AI 在系统里承担了什么角色？
- 人类控制点在哪里？
- 产生了哪些可审计证据？
- 失败和边界是什么？
- 下一个阶段为什么成立？

## 叙事主线

```text
个人 AI 助手
  -> 本地工程 Loop
  -> 远端任务控制台
  -> 多角色 AI 协作
  -> 可治理的 AI 工作系统
```

我们要讲清楚的不是“AI 更聪明了”，而是“AI 被放进了一个可治理的工作系统”。

分享时必须先讲 `09-north-star.md` 的终局效果，再讲 FUZ-554 的阶段案例。否则听众容易把这个项目误解成一组脚本，而不是一套 AI 工程控制结构。

终局表达：

- Multica 是任务事实源。
- ai-loop 是本地执行事实源。
- Multica Loop 是组织和治理层。
- Agent Network 是能力角色层。
- Artifacts & Memory 是知识沉淀层。

最终演示不是“AI 写完代码”，而是一个 issue 从进入、拆解、执行、举证、复核、回写建议到经验沉淀的完整链路。

## 阶段分享设计

### 分享 1：为什么需要 AI 工作编排

核心观点：单个 Agent 能提效，但没有闭环就无法成为团队能力。

建议内容：

- North Star：最终要做的是 AI 工程团队操作系统，而不是脚本集合。
- 当前痛点：AI 执行快，但证据散、状态散、复盘难。
- 目标：从一次性对话转向可追踪工作流。
- 架构：Multica 控制台 + ai-loop 执行内核 + artifacts 事实层。
- 原则：Local first、Evidence first、Human in command。

输出文档：

- `09-north-star.md`
- `00-vision.md`
- `01-domain-model.md`

### 分享 2：Phase 1 MVP 复盘

核心观点：先打通单 issue 安全链路，而不是追求全自动。

建议内容：

- 如何从 `FUZ-xxx` 生成本地 task。
- 为什么先 dry-run。
- artifacts 如何成为事实来源。
- comment 草稿如何帮助人类决策。
- 哪些动作被故意禁止。

输出文档：

- `02-phase1-mvp.md`
- `logbook.md`
- 首个试点 issue 的案例复盘。

### 分享 3：从 dry-run 到真实执行

核心观点：真实执行的前提不是“模型可用”，而是验收标准可验证。

建议内容：

- repo、verify command、approval policy 如何进入任务。
- Loop 如何创建 worktree。
- Codex 如何执行。
- verify 如何失败并触发下一轮。
- patch 和 summary 如何被复核。

输出文档：

- 真实 run 记录。
- verify 失败与重试案例。
- patch 复核案例。

补充入口：

- `README.md`：总入口索引。
- `05-case-playbook.md`：低风险案例执行指南。
- `cases/FUZ-554/README.md`：首个可复盘案例。

### 分享 4：多角色协作

核心观点：不同智能体不是“多个聊天窗口”，而是不同职责边界。

建议内容：

- Codex-顾实：主力开发。
- Xcode：二意见和复杂判断。
- 黑墙：调度和总控。
- 测真：验证复核。
- 人类：目标、边界、验收和最终决策。

输出文档：

- 角色分工手册。
- 任务路由规则。
- 失败升级规则。

### 分享 5：治理与规模化

核心观点：规模化的关键不是更多自动化，而是更清晰的策略和边界。

建议内容：

- 批量队列如何准入。
- 哪些状态可以自动改，哪些必须人工确认。
- 如何防止重复执行。
- 如何衡量 AI 工作质量。
- 如何管理理解债务。

输出文档：

- 策略配置说明。
- 指标看板草案。
- 风险清单。

## 每次推进的固定记录格式

每次推进都写入 `logbook.md`：

```text
## 日期：阶段 / 动作

### 目标

### 决策

### 执行

### Evidence

### 问题

### 下一步
```

## 衡量指标

短期指标：

- 从 issue 到 task 的转换成功率。
- dry-run 成功率。
- task 是否包含 repo、验收、验证命令。
- summary 是否能支撑人工判断。

中期指标：

- 真实 run 通过率。
- verify 失败后的自动修复成功率。
- 人工介入点数量。
- 从 issue 到 in_review 的时间。

长期指标：

- 团队可复用任务模板数量。
- AI 产出被人工接受比例。
- 失败复盘沉淀数量。
- 关键项目中 AI Loop 覆盖比例。

## 分享时要避免的误区

- 不要把它讲成“自动做需求”。
- 不要把模型能力当成系统能力。
- 不要忽略失败案例。
- 不要弱化人工控制点。
- 不要把状态同步当成完成证明。
