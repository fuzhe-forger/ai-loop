# 司南 v1.1 到 v2.0 路线图

## 1. 路线图口径

这份路线图不是 v1.0 的验收条件，而是 v1.0 released 之后的正式演进规划。它用于回答：司南接下来从“本地治理能力可验收”走向“稳定可复制的 AI 需求交付平台”，还需要补哪些能力、按什么顺序补、每一阶段如何验收。

## 2. 北极星目标

到 v2.0，用户给出一个目标后，司南能够主动判断任务规模和适用性，选择合适流程，组织本地执行与必要的外部写回，并把全过程沉淀为可审计证据。人类仍保留最终业务验收、部署、远端决策和高风险副作用审批权。

## 3. 当前基线：v1.0

| 维度 | v1.0 状态 |
|---|---|
| 版本 | `v1.0.0 released` |
| 形态 | 本地 CLI + Markdown 协议 + 飞书/Multica/Obsidian 写回 |
| 已有能力 | 门禁、计时、evidence、受控写回、记忆、多角色、分享包、能力注册表 |
| 核心限制 | 缺少产品化入口、真实 E2E 样本少、估时样本少、跨仓库与服务化不足 |
| 禁止越界 | 不自动部署、不自动生产操作、不自动 reviewer 裁决、不自动远端写回决策 |

## 4. 阶段总览

| 版本 | 主题 | 目标 | 验收标准 |
|---|---|---|---|
| v1.1 | 产品化入口与真实 E2E | 让新人能用，让真实任务能复用 | 一页式手册、doctor、3 类 E2E case pack 完成 |
| v1.2 | 估时回归与任务分级 | 让“是否启用司南、跑多久、何时停”更准 | 任务分级稳定，估时 1 分钟命中率达到 60% |
| v1.3 | 轻重流程自适应 | 简单任务不被重流程拖慢，复杂任务不漏门禁 | L0-L4 gate policy 可自动建议，人工可覆盖 |
| v1.4 | 记忆与交接升级 | 减少重复解释和长会话污染 | handoff、memory review、去重、推荐命中率可量化 |
| v1.5 | 外部系统适配增强 | 飞书/Multica 写回更稳，失败可恢复 | schema mapping、readback、失败恢复指南可用 |
| v1.6 | 多仓库与 CodeGraph 编排 | 支持跨仓库任务定位、证据聚合 | 至少 2 个 repo 的跨仓库案例跑通 |
| v1.7 | 团队运营看板 | 让管理者看见提效、返工、风险 | 估时误差、gate 失败、写回、返工指标可生成 |
| v1.8 | 半自动策略建议 | 系统能建议 gate 强度、审批范围、下一步 | 建议可解释，人类确认后执行 |
| v1.9 | 平台化封装 | 从脚本集合走向稳定 CLI/API | CLI/API、配置、状态目录、权限模型稳定 |
| v2.0 | 可复制 AI 需求交付平台 | 多项目可接入，流程稳定可审计 | 真实项目复制成功，v2.0 验收套件通过 |

横向课题：`sinan-token-efficiency-loop-plan.md` 覆盖 v1.2、v1.4、v1.7，目标是把 Token 使用率从“个人习惯”升级为“可度量、可治理、可复盘”的系统能力。

## 5. v1.1：产品化入口与真实 E2E

### 目标

解决“文档够工程但不够产品化”“真实业务案例不够”的问题。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| 一页式产品介绍 | 面向新人解释司南是什么、适合什么、不适合什么 | 飞书文档可直接转发 |
| 快速上手手册 | 10 分钟跑通一个低风险任务 | 新人按手册可完成 dry-run |
| 真实文档任务 E2E | 从需求到文档产出、飞书写回、readback | 生成 case pack |
| 真实代码任务 E2E | 从方案到 patch、验证、review packet | 生成 patch evidence |
| 真实写回任务 E2E | 飞书/Multica 写回和 readback 完整闭环 | 生成 writeback evidence |
| 团队接入 doctor | 检查环境、脚本、权限、skills、配置 | 一条命令输出 ready/not ready |

### 优先级

P0：产品介绍、快速上手、三类 E2E。
P1：doctor、案例模板。

## 6. v1.2：估时回归与任务分级

### 目标

把当前估时从“事后复盘”升级为“可用于任务分片的回归模型”。

### 当前问题

- 当前闭合样本少。
- 1 分钟命中率约 33.3%。
- 近期主要偏差是高估，而不是低估。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| 估时样本池 | 每个切片记录 task_type、估时、实际、误差 | 样本数 >= 30 |
| 任务分级模型 | L0-L4 按风险、规模、副作用、验证成本分层 | 分类命中率人工抽检通过 |
| 估时桶 | 文档、脚本、写回、验证、发布等桶独立校准 | 每桶至少 3 个可信样本 |
| 误差看板 | 输出 MAE、最大误差、1 分钟命中率 | 指标可生成 markdown/json |
| 下次估时建议 | 自动输出下次建议区间 | 建议进入 closeout |
| Token 使用率采样 | 记录长会话、重复读取、冗长汇报和 evidence 复制问题 | 形成 token baseline report |

### 目标指标

- v1.2：1 分钟命中率 >= 60%。
- v1.3：1 分钟命中率 >= 70%。
- v2.0：常见任务 1 分钟命中率 >= 80%。

## 7. v1.3：轻重流程自适应

### 目标

让系统主动判断任务规模，避免简单任务被重流程拖慢，也避免复杂任务漏门禁。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| L0/L1 快速通道 | 简单问答、轻量文档只走最小证据 | 不强制完整 Loop |
| L2 标准流程 | 普通脚本/文档/写回走标准门禁 | 产出 evidence 和 closeout |
| L3 深度流程 | 多文件、多系统、有风险任务走方案门禁和 review | 必须有 design gate |
| L4 暂停审批 | 部署、生产、删除、远端发布必须暂停 | 不允许自动执行 |
| 流程建议器 | 任务开始时主动建议是否启用司南 | 用户可一键接受/覆盖 |

## 8. v1.4：记忆与交接升级

### 目标

解决长会话污染、重复解释、经验不复用的问题。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| Handoff 标准 | 长任务阶段性沉淀交接文档 | 新会话可从 handoff 继续 |
| Token-efficient handoff | 交接文档只保留状态、证据路径、决策、下一步 | 不复制完整历史上下文 |
| 记忆质量检查 | 检查重复、过期、敏感信息 | memory quality report 通过 |
| 候选记忆提取 | 从 evidence 提取经验候选 | 默认需人工确认 |
| 记忆推荐 | 任务开始推荐相关经验 | 推荐命中率可抽检 |
| 记忆老化 | 过期经验降权或归档 | 不自动删除 |

## 9. v1.5：外部系统适配增强

### 目标

让飞书/Multica 写回更稳定，不再依赖单一表结构或临时路径。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| 飞书 doc/base schema mapping | 识别文档、表格、字段、视图差异 | 可生成写回计划 |
| Multica issue adapter | 标准化 issue/comment/status/metadata 写回 | 可 readback |
| 写回失败恢复 | 网络、权限、字段错误可诊断 | 输出 retry/人工处理建议 |
| 写回审计包 | 每次写回都有 before/write/readback | evidence 完整 |
| 只读预检 | 写入前先检查目标和权限 | 降低误写风险 |

## 10. v1.6：多仓库与 CodeGraph 编排

### 目标

从单仓库治理升级到多仓库任务定位和证据聚合。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| 仓库识别 | 根据任务自动识别涉及 repo | 识别结果可人工确认 |
| CodeGraph 定位 | 使用代码结构快速定位影响面 | 减少盲搜 |
| 跨仓库 evidence | 多 repo 结果聚合到一个 run | review packet 可读 |
| 分支流程 skill | release/pre/staging 标准流程技能化 | 仅必要时用 LLM |
| 冲突边界 | 多仓库冲突、远端操作必须审批 | 无自动 push/deploy |

## 11. v1.7：团队运营看板

### 目标

让司南从“能执行”升级为“能管理和改进”。

### 关键指标

| 指标 | 说明 |
|---|---|
| 需求数 | AI 参与的业务需求数量 |
| 上线/交付节点 | AI 支持到可交付或上线的节点数 |
| 提效比例 | 分环节估算节省时间 |
| 估时误差 | MAE、最大误差、1 分钟命中率 |
| Gate 失败原因 | 需求不清、方案缺失、证据不足、写回越界 |
| 返工率 | AI 初稿后小改引入问题的比例 |
| 人类卡点 | AI 加速后流程回到人侧的等待点 |
| Token 使用率 | 输出密度、重复读取率、evidence 引用率、handoff 复用率 |

## 12. v1.8：半自动策略建议

### 目标

让系统根据历史证据给建议，但不替代人类决策。

### 可自动建议

- 任务等级。
- 是否启用司南。
- 需要哪些 gate。
- 预估耗时区间。
- 是否需要飞书/Multica 写回审批。
- 下一步最小可验收切片。

### 不自动决策

- reviewer 最终 approve/request changes。
- 远端写回最终决策。
- 部署和生产操作。
- 删除、归档、权限变更。

## 13. v1.9：平台化封装

### 目标

减少脚本散落，把协议沉淀为稳定 CLI/API。

### 关键能力

| 能力 | 说明 | 验收 |
|---|---|---|
| `sinan` CLI | 封装 plan/run/closeout/writeback/status | 一套入口可用 |
| 配置目录 | 统一 config、policy、adapter | 可迁移 |
| 状态目录 | 统一 runs、memory、evidence | 可审计 |
| 插件接口 | Feishu/Multica/Obsidian/CodeGraph 可插拔 | adapter contract 稳定 |
| 权限模型 | 本地/外部/部署/生产边界清楚 | L4 必停 |

## 14. v2.0：可复制 AI 需求交付平台

### 目标

司南可以被复制到多个项目/团队，稳定支撑 AI 参与需求交付。

### v2.0 验收标准

| 标准 | 验收方式 |
|---|---|
| 新人 30 分钟内理解并跑通低风险任务 | onboarding 演练 |
| 三类真实 E2E 完整 | 文档、代码、写回 case pack |
| 常见任务估时 1 分钟命中率 >= 80% | timing dashboard |
| 外部写回 100% 有 readback | writeback audit |
| 无审批部署 0 次 | approval boundary audit |
| 多仓库任务可聚合 evidence | cross-repo case |
| 团队复盘可看到提效和返工数据 | ops dashboard |

### 本地验收套件

v2.0 本地验收由 `scripts/sinan-v2-acceptance.sh` 承载，覆盖产品手册、路线图、onboarding drill、Token 使用率治理、flow advisor、ops dashboard、多仓库 evidence、外部 adapter 预检、v2 gap audit 和工具链验证。该套件只证明本地平台能力闭环，不代表已完成外部写回、真实部署或团队规模化 adoption。

## 15. 近期建议执行顺序

### 第一阶段：先补产品化入口

1. 固化 v1.0 产品说明与使用手册。
2. 补一页式介绍。
3. 做新人快速上手 dry-run 示例。
4. 更新飞书路线文档索引。

### 第二阶段：补真实 E2E

1. 文档型 E2E。
2. 代码型 E2E。
3. 写回型 E2E。
4. 每类沉淀 case pack。

### 第三阶段：估时回归

1. 收集 30 个可信 timestamp 样本。
2. 做任务类型估时桶。
3. 建 timing dashboard。
4. 把高估/低估原因写入 closeout。

### 第四阶段：轻重流程自适应

1. 定义 L0-L4 策略。
2. 简单任务快速通道。
3. 复杂任务完整 gate。
4. L4 副作用强制暂停。

## 16. 风险与取舍

| 风险 | 取舍 |
|---|---|
| 过早平台化 | 先用文档和脚本稳定协议，再封 CLI/API |
| 自动化越界 | 先建议，后审批，不做自动最终决策 |
| 记忆污染 | 候选记忆默认人工确认 |
| 看板维护成本 | 先生成 markdown/json，再考虑 UI |
| 估时过拟合 | 样本少时只做轻权重修正 |

## 17. 暂缓清单

以下能力在 v2.0 前仍默认暂缓或仅做建议，不做自动执行：

- 自动 reviewer 最终裁决。
- 自动远端写回决策。
- 无审批部署。
- 生产系统自动操作。
- 未确认长期记忆自动写入。
- 未确认的删除、归档、权限变更。

## 18. 总结

v1.0 证明司南能把 AI 需求交付纳入治理闭环。v1.1 到 v2.0 的核心不是追求更“自动”，而是让它更好用、更准、更稳、更容易复制：产品化入口、真实 E2E、估时回归、轻重流程自适应、记忆交接、多仓库编排、运营看板和平台化封装。
