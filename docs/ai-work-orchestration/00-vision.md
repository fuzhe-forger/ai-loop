# AI 工作编排实践：愿景与定位

## 一句话定位

我们要建设的不是一个 issue 同步脚本，而是一套可控、可验证、可复盘的 AI 工作编排实践。

Multica 是远端控制台，ai-loop 是本地执行内核，Codex、Claude、OpenClaw 是不同类型的执行与判断能力。三者联动后，一个任务可以从想法进入系统，经过判断、执行、验证、反馈，再进入下一轮决策。

## 为什么要做

单个 AI Agent 可以完成很多局部任务，但真实工程工作不是一次回答就结束的。它需要：

- 明确目标和边界。
- 在隔离环境里执行。
- 产生可审计证据。
- 失败后能复盘和重试。
- 人类可以随时接管。
- 团队成员能理解发生了什么。

如果没有这套闭环，AI 很容易变成“看起来很快，但不可控”的黑盒执行器。

我们的目标是把 AI 从一次性助手，推进到工程系统中的稳定工作单元。

## 核心判断

这件事的核心不是自动化程度，而是控制结构。

我们不追求让 AI 任意接管任务，而是追求让每一次 AI 执行都具备以下特征：

- 有明确输入。
- 有执行边界。
- 有本地隔离。
- 有验证命令。
- 有产物证据。
- 有状态反馈。
- 有人工决策点。

只有具备这些条件，AI 才能从“个人效率工具”变成“团队工程能力”。

## 系统分层

```text
Multica 控制台
  负责任务、状态、负责人、协作、人工决策

黑墙 / 调度策略
  负责判断任务该如何处理、派给谁、什么时候停

ai-loop 本地执行内核
  负责 worktree、Codex 执行、验证、重试、summary、patch

Codex-顾实 / Xcode / 测真等能力角色
  负责开发、二意见、测试、文档、复核

Artifacts 事实层
  保存 task、run、prompt、日志、diff、verify、summary
```

## 我们坚持的原则

### 1. Evidence first

Issue 状态不是完成证明。真正的完成证明来自本地 artifacts：summary、patch、verify 结果、日志和人工复核结论。

### 2. Local first

真实工程执行优先发生在本地隔离 worktree。默认不 push，不建 MR，不部署，不写远端状态。

### 3. Human in command

人类不是退出系统，而是从执行者上移为目标、边界、验收和决策的设计者。

### 4. Explicit side effects

任何外部写入都必须显式授权，包括 Multica comment、状态变更、远程 Git、飞书写入、部署等。

### 5. Small loop before big loop

先打通单个 issue 的闭环，再做批量队列、智能调度和多智能体协作。

## 先锋实践价值

这个方向可以沉淀为团队级 AI 工程实践：

- 让 AI 执行过程可审计。
- 让复杂任务具备可复盘链路。
- 让不同智能体角色有稳定分工。
- 让失败不再是黑盒，而是下一轮输入。
- 让团队可以分享、复制和持续改进这套工作方式。

## 第一阶段目标

第一阶段只打通主链路，不追求全自动：

```text
Multica Issue
  -> Work Item
  -> ai-loop Task
  -> local dry-run / run
  -> Evidence
  -> Multica Feedback 草稿
```

第一阶段成功的标志不是“自动做完所有事”，而是：我们能把一个远端 issue 安全、稳定地转成本地 Loop 任务，并产生可分享、可审计的执行证据。
